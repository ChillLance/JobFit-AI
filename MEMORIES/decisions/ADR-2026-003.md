# ADR-2026-003: Replace fixed career-direction discovery with a time-bound Search Mission

- Status: accepted 2026-07-16 (owner-directed, implemented same day; this ADR
  records it — the underlying design reasoning already lived in
  `MEMORIES/handoff-2026-07-16-search-mission.md`)
- Scope: profile-direction UX, job-decision surfacing, discovery prompts
- Supersedes: the "direction discovery" half of ADR-2026-001 / the
  Base-Bridge-Target flow shipped under ADR-2026-002's predecessor work
  (`SearchIntent` type, `DIRECTION_DISCOVERY_PROMPTS`) — the *four-layer job
  evaluation model* and *human-confirmed submission* guardrails from
  ADR-2026-001 are unaffected and still binding.

## Decision

Replace the fixed career-direction hypothesis (Base/Bridge/Target, meant to
converge on one confirmed direction) with a **Search Mission**: a time-bound,
possibly-plural job-search purpose that stays explicitly separate from the
stable Profile.

Rationale, from user #0 evidence
(`MEMORIES/user0-retrospective-2026-07.md`):
- Stable facts (visa, language, verified experience) are entered once;
  what changes between searches is the *purpose* of this particular search
  (season, experience goal, growth goal, region, tradeoffs) — collapsing
  both into one "confirmed direction" forced re-deciding identity every time
  the actual need was just "what am I doing this round for."
- Seasonal/regional experiences (e.g. a winter-only ski job) are real and
  valid but must not silently become a permanent Profile preference.
- Multiple directions can coexist honestly (1-3 concurrent missions) rather
  than being forced into a single confirmed lane.
- Job source does not identify agency vs. direct — agency detection must
  come from structured extraction/listing evidence, never from which site a
  listing was collected from.

Implementation: `src/lib/missions/` (`SearchMission` schema, `evaluateMission`
transparent rule engine, local store), `/missions` page, `SEARCH_MISSION_DISCOVERY_PROMPTS`
in `profilePrompt.ts`. Decision output (`apply` / `confirm` / `not_recommended`)
is shown beside, and never overwrites, the existing rule-based Fit Score.

## Cleanup performed under this ADR (2026-07-16, same day)

The superseded design left dead code that is now removed (verified zero
remaining references before deletion):
- `SearchIntent` / `SearchIntentLane` types and the unused
  `JapanCareerProfile.searchIntent` field (`src/lib/profile/types.ts`).
- `DIRECTION_DISCOVERY_PROMPT_ZH/EN/JA` and `DIRECTION_DISCOVERY_PROMPTS`
  (`src/lib/profile/profilePrompt.ts`), plus the test pinning them
  (`profilePrompt.test.ts`).

## Consequences

- Positive: the onboarding question now matches how this user (and
  presumably others in the same seasonal-work pattern) actually thinks about
  a job search; agency-vs-direct question timing is handled correctly by
  route inference instead of by source site.
- Positive: directly enables the Application Kit slice named in
  ADR-2026-002 (Mission goals feed the application draft's motivation text).
- Negative: single-user evidence again (same caveat as ADR-2026-002) — Slice
  2 (3-5 external users) should confirm the Mission framing generalizes
  before more UI is built on top of it.
- The decision-rule guardrails (unknown → confirm, not a guess; explicit
  structured blockers only; free-form goals stay visible, never silently
  scored) follow the same "missing info is a question, not a penalty"
  principle as ADR-2026-001's four-layer model and `localAnalysis.ts` — this
  is a consistent extension of that principle, not a new one.
