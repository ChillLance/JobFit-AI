每份模板都內建派工三件套（目標與動機／驗收條件／回報格式）。共通規則：
- 回報上限 500 字；長產物落檔傳路徑（model-dispatch.md §4）。
- 「找不到」就回「未發現」，禁止編造。
- Windows 環境：路徑用反斜線或正斜線皆可，但 PowerShell 語法限制見 CLAUDE.md 本機陷阱。

## T1 搜尋（Explore；≥10 檔用 sonnet，≤3 個明確位置的 read-back 用 haiku）

```
【目標與動機】我在做【任務一句話】，需要先知道【要找什麼】在哪裡。你只負責找，不負責改。

【範圍】在【目錄／檔案模式，例 src/**/*.ts】裡找【具體 pattern 或概念】。
禁止讀 node_modules；單檔 >1000 行只讀命中段落前後 20 行。

【驗收條件】每個結果附 檔案:行號；自認找完後，換一個同義詞或不同命名慣例再掃一輪才准回報。

【回報格式】條列，每行「檔案:行號 — ≤20 字說明」。超過 30 筆就寫到【scratchpad 路徑】，回報只給路徑＋總數＋前 5 筆。沒找到明說「未發現」。
```

## T2 實作（general-purpose；預設 sonnet）

```
【目標與動機】為什麼要做：【動機一句話】。要做什麼：【具體功能／改動】。

【規格】
- 改動範圍限這些檔案：【清單】。範圍外的檔案發現問題只回報、不要動。
- 遵循慣例：先讀【1-2 個同型現有檔案】照它的寫法（「同型」＝同目錄下職責相同的檔案；例：要加 API route 就先讀一個既有的 `src/app/api/**/route.ts`）。
- 流程鐵律：每次 Edit 前先 Read 目標檔案的相關段落（違者工具會直接報錯）。

【驗收條件】
1. `npx tsc --noEmit`、`npm run lint`、`npm test` 三者 exit code 0（把每個指令輸出的最後一行抄進回報）。
2. 【功能專屬驗收，例：新函式對【輸入】回傳【輸出】，有對應測試】。

【回報格式】≤500 字：改了哪些檔（檔案:行號）、三個檢查的結果行、你做過的假設。diff 不用貼。
```

## T3 除錯（general-purpose；預設 sonnet；此模板 prompt 第一行的計畫指令是替代「effort 參數」的手段）

```
先列出你的診斷計畫（≤5 步），再逐步執行。

【目標與動機】症狀：【錯誤訊息原文／異常行為】。觸發方式：【怎麼重現】。我需要根因和修法，不是繞過。

【已知線索】【已試過什麼、結果如何——沒有就寫「無」】

【規則】
- 先重現，重現不了就停下回報，不准盲修。
- 同一個假設驗證失敗 2 次就換假設；總共 2 輪失敗就停下，把失敗軌跡（試了什麼→輸出什麼→為何不符預期）完整寫進回報。
- 修好後必須說明「為什麼之前會壞」——說不出根因的修復不算修復。

【驗收條件】重現步驟執行後症狀消失＋`npm test` exit 0＋根因一句話。

【回報格式】≤500 字：根因、改動（檔案:行號）、驗證證據。失敗則回報失敗軌跡（這是有效產出，不是丟臉）。
```

## T4 重構（general-purpose；預設 sonnet；跨 ≥4 檔的結構決策先派 Plan＋最強模型出計畫）

```
【目標與動機】重構【對象】因為【動機】。行為不准變，結構要變成【目標形狀】。

【規則】
- 開工前先跑 `npm test` 記錄基線（必須全綠才准開工；有紅的先停下回報）。
- 小步走：每完成一個可編譯的中間狀態就跑一次 `npx tsc --noEmit`。
- 禁止順手改行為、順手改格式、順手升依賴。

【驗收條件】`npm test` 結果與基線完全相同＋`npx tsc --noEmit`、`npm run lint` exit 0＋【結構驗收，例：X.ts 不再 import Y】。

【回報格式】≤500 字：動了哪些檔（檔案:行號）、測試基線前後對照（數字）、沒動什麼（明列範圍外發現的問題）。
```

## T5 研究（general-purpose 或 Explore；預設 sonnet；WebSearch/WebFetch 需先 ToolSearch 載入）

```
【目標與動機】我要決定【決策】，需要你查清楚【問題】。結論會直接影響【後續動作】。

【範圍】查【來源：官方文件／repo 內／指定網址】。官方文件優先於部落格；註明每條資訊的來源與日期。
提醒：WebSearch / WebFetch 是 deferred tools，先用 ToolSearch "select:WebSearch,WebFetch" 載入。

【驗收條件】每個結論附來源連結或 檔案:行號；查不到的部分明說「查無」，並區分「我沒查到」與「官方明說不支援」。

【回報格式】結論先行（≤3 句），然後條列證據。全文 ≤500 字；更長的整理寫到【scratchpad 路徑】。
```

## T6 審查（Explore＝唯讀，防止審查者順手改碼；預設 sonnet；高風險改動用最強模型）

```
【目標與動機】審查【diff 範圍／檔案清單】。這段改動的目的是【原任務一句話】。你是 fresh reviewer，之前的討論你不知道也不需要知道。

【審查重點（按序）】
1. 正確性：邏輯錯誤、邊界條件、錯誤處理缺口。
2. 範圍：diff 是否含與目的無關的改動。
3. 造假訊號：有沒有 test.skip / eslint-disable / ts-ignore / 放寬斷言來讓檢查通過。
4. 專案規則：對照 AGENTS.md 硬規則（'use client' 禁令、不加依賴、不動 jobs_temp.json）。

【驗收條件】每個 finding 附 檔案:行號＋一句失敗情境（什麼輸入會出什麼錯）。不確定的標【存疑】。零 finding 就明說「未發現問題」，不准硬湊。

【回報格式】按嚴重度排序條列，每條 ≤2 行。≤500 字。
```

- **校準備註**：同一天、同一個 haiku，**無模板**時連「數 47 行檔案裡的 5 個標題」都數錯（見 environment.md §3）；**套模板**（內建「換同義詞重掃一輪」的驗收步驟）後結果全對。結論：低階模型的可靠度大半來自模板裡的強制驗證步驟——所以【驗收條件】欄不准留空、不准刪。
