## 1. 何時該升級模型（補充 model-dispatch.md §5 的錯誤次數門檻，這裡是「開工前」的判斷）

**判準**：任務滿足以下任一條，開工前就直接用最強模型，不要先讓便宜模型試錯：
(a) 問題是「為什麼」而不是「是什麼」（因果推理，非查找）；
(b) 需要同時把 ≥4 個檔案的內容放在腦裡做取捨；
(c) 錯一次的代價 > 重做十次的代價（動使用者資料、對外發布、改公開 API）。

- **正例**：「分析為什麼 analyze 路由偶爾回傳舊 profile」——因果推理＋跨 route/repository/sync 多檔——直接最強模型。
- **反例**：「把所有 console.log 換成 logger」就先升 opus——這是機械工作，sonnet 甚至派給 haiku 批次做即可；升級是浪費。

## 2. 何時算「真的完成」

**判準**：以下五格全部打勾才准說完成，缺任何一格只能說「進度回報」：
(1) 使用者原話的**每一個子句**都有對應交付（把原話逐句抄下來對照）；
(2) `npm test`、`npx tsc --noEmit`、`npm run lint` 三者 exit code 0（純文件改動可略過，但要明說略過了什麼）；
(3) 產出已落檔，不是只存在對話裡；
(4) 驗收由 fresh-context agent 完成，**或**由機器判定（測試／typecheck 的 exit code——判定者是框架，不是寫碼者的眼睛；界線見 model-dispatch.md §6），總之不是自己看自己說好；
(5) diff 裡沒有任務範圍外的檔案。

- **正例**：「已完成：三個檢查 exit 0（貼輸出末行）、新增 salary.test.ts 覆蓋兩個邊界、read-back 確認 salary.ts 匯出簽名與規格一致。」
- **反例**：「程式碼寫完了，應該可以用」——沒跑測試、沒驗收，這是進度回報卻宣告了完成。歷史教訓：宣告完成後被使用者發現沒動到需求的第二子句，返工比一開始慢做更貴。

## 3. 何時該停下來問使用者

**判準**：滿足任一條就停下來問，其餘情況一律「記錄假設、繼續做」：
(a) 動作不可逆**且**無法先備份（可備份就先備份再做，不用問）；
(b) 需求存在兩種合理解讀，且兩種解讀的交付**不同檔案或不同行為**（同檔案小差異就選較保守的做）；
(c) 要賭一個未確認假設超過 30 分鐘工時或 3 次派工；
(d) 品味類決策（視覺風格、配色、文案語氣、命名偏好）——見 §6。

- **正例**：「要我把 profile 頁改成 server-fetched 嗎？這會改變 localStorage 行為（REDESIGN.md 明文保留它）——兩種做法交付不同，先問。」
- **反例**：「請問測試檔要放哪個目錄？」——repo 裡現有 6 個 `*.test.ts` 全在 `src/lib/**` 同層，慣例可查，不該問。問得出口的問題，八成環境裡就有答案。

## 4. 什麼訊號代表方向錯了（該換路，不是再試一次）

**判準**：出現任一訊號，立刻停手，回到最近一個「全綠」狀態（測試全過的 commit 或對話節點），重新拆問題：
(a) 同一個測試連 2 輪修改後還是紅的；
(b) 每修好 1 處，冒出 ≥2 處新錯誤；
(c) 發現自己在改「和任務無關的檔案」來讓測試過；
(d) 在跟型別系統或框架搏鬥（cast、any、ts-ignore 越加越多），而不是在跟問題搏鬥；
(e) 對同一個錯誤訊息第 3 次做出「再跑一次看看」的決定（diagnosis.md D8b：歷史上重試 10 次零成功）。

- **正例**：修 hydration error 兩輪失敗後，停手改查 next-best-practices skill 的 hydration-error.md，發現是 server/client 時間戳差異——換路 5 分鐘解決。
- **反例**：`preview_screenshot` 逾時就再呼叫一次，共 10 次（真實發生於 2026-07-02）——正確做法是第 2 次逾時就改用 preview_snapshot 或重啟 dev server。

## 5. 品質底線怎麼驗（最低可接受標準，不是理想標準）

**判準**（程式碼類任務全數必過）：
(1) 三個檢查 exit 0（同 §2）；
(2) 新增的每個匯出函式至少 1 個對應測試；
(3) UI 改動附截圖證據；
(4) 回報敘述與實際 diff 一致（說改了 A 就真的只改了 A）；
(5) 沒有為了過檢查而放寬檢查（禁止加 eslint-disable / ts-ignore / test.skip 來「通過」，除非使用者同意）。

- **正例**：加了 `getSalaryRange()`，同 commit 附兩個測試（正常區間＋空字串），截圖免（無 UI）。
- **反例**：測試紅了就 `test.skip` 掉然後回報「測試通過」——這是造假，比測試紅更嚴重。

## 6. 品味與模糊題＝harness 的極限（誠實條款）

拆解、驗證、多樣本評審補得了**執行品質**；補不了**品味與模糊需求**。證據：diagnosis.md D8d——歷史 transcript 裡唯一的使用者強烈糾正就是配色（「一堆 AI 做出來的產品都在用的顏色」），功能類零糾正。

**遇到品味類任務（視覺、文案語氣、命名、「好不好看」「順不順」）的固定做法**：
1. 不要單方案賭。第一版就給 **2–3 個方向**（各附一句差異說明或截圖）讓使用者挑。
2. 使用者挑了之後，把選擇寫進 DESIGN.md 或 memory（按 maintenance.md 格式），下次不用再問。
3. 如果使用者不在線且非要往下做：選**最不像預設值**的方向（歷史糾正就是「太像大家的預設」），並在回報裡標明「品味決策未經確認」。

**遇到模糊需求**：能查環境就查；查不到就把兩種解讀各用一句話寫出來，按 §3(b) 判斷要不要問；決定不問就選保守解讀並在回報開頭標明假設。**禁止**假裝需求是清楚的。
